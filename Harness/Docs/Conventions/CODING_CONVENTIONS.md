# 코딩 컨벤션

## 언어 / 프레임워크

- C++
- raylib

## 네이밍

- 클래스 / 구조체: PascalCase
- 함수 / 메서드: PascalCase 또는 기존 파일 스타일 유지
- 멤버 변수: 프로젝트에 맞게 결정
- 상수: PascalCase 또는 기존 파일 스타일 유지
- 파일명: 기존 파일 스타일 유지

## 주석

- 코드만으로 의도가 분명하면 주석을 추가하지 않는다.
- 업데이트 순서, 좌표 계산, 렌더링 기준처럼 맥락이 필요한 부분에만 짧게 남긴다.
- 오래된 주석은 코드 변경 시 함께 갱신한다.

## 구조

- 게임 루프 책임은 `Game`에 둔다.
- 씬별 구성은 `src/scenes`에 둔다.
- 게임 엔티티는 `src/entities`에 둔다.
- UI 배치와 위젯은 `src/ui`에 둔다.
- 크기 조절, 스테이지 기믹, 출구 조건처럼 게임 규칙에 가까운 기능은 엔티티와 씬 책임을 분리해서 배치한다.
- 플레이어 1/2의 공통 동작은 중복 구현하지 않고 같은 코드 경로를 우선 사용한다.
- 새 기능은 기존 책임 경계를 먼저 확인한 뒤 배치한다.

## raylib 사용

- 이미지 포맷은 PNG/BMP를 우선 사용한다.
- 오디오는 WAV/OGG를 우선 사용한다.
- 3D 모델, 외부 폰트, JPG/GIF/MP3 등은 현재 프로필에서 제한될 수 있으므로 사용 전 확인한다.
- 저용량 구조를 유지하기 위해 새 에셋과 기능 추가 시 파일 크기와 빌드 프로필 영향을 확인한다.

## 작업 로그

- 코드 변경 후 `FILE_CHANGE_LOG.md`에 변경 이력을 남긴다.
- 로그는 긴 설명보다 파일명, 이유, 검증 여부 중심으로 작성한다.
