# AI 개발 규칙

## 작업 우선순위

1. 현재 대화의 직접 프롬프트
2. `Harness/WORK_ORDER.md`
3. 하네스 문서와 프로젝트 구조

충돌이 있으면 더 높은 우선순위의 지시를 따른다.

## 작업 원칙

- 작업 전 관련 파일을 먼저 읽는다.
- 프로젝트에 이미 있는 구조와 스타일을 우선한다.
- 확인되지 않은 정보는 지어내지 않고 `확인 필요`로 남긴다.
- 사용자 또는 다른 작업자가 만든 수정사항을 임의로 되돌리지 않는다.
- 현재 작업과 관련 없는 리팩터링은 피한다.

## 구현 원칙

- 기존 public API와 책임 경계를 불필요하게 흔들지 않는다.
- `Game`, `Scene`, `GameObject`, `Player`, `ui`의 역할을 분리해서 유지한다.
- SIZE UP의 핵심인 크기 조절, 2인 협동, 짧은 스테이지 반복 플레이를 우선 고려한다.
- 플레이어 2명은 가능하면 동일한 행동 능력을 공유하게 설계한다.
- 크기 변화는 충돌 범위, 이동 가능 공간, 발판 활용과 연결될 수 있게 확장성을 남긴다.
- raylib 커스텀 빌드에서 꺼진 기능은 사용하기 전 `RAYLIB_PROFILE.md`를 확인한다.
- 에셋은 현재 지원되는 포맷을 우선 사용한다.

## 검증 원칙

- 가능한 경우 빌드 또는 실행 검증을 수행한다.
- 검증하지 못한 경우 이유를 작업 결과에 명시한다.
- 플레이어 이동, 크기 단계, 충돌, 출구 조건을 수정한 경우 최소 수동 검증 항목을 남긴다.
- 시각 변경은 창 표시, UI 잘림, 픽셀 아트 선명도, 에셋 로딩을 확인한다.

## 작업 기록 규칙

- 작업 완료 후 `Harness/Docs/Logs/FILE_CHANGE_LOG.md`를 갱신한다.
- 변경한 파일, 변경 이유, 검증 여부를 짧게 기록한다.
- 미확인 또는 남은 일은 별도로 표시한다.
